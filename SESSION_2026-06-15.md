# FFTT04L — Session changes (2026‑06‑14 → 06‑15)

Branch **main**. All items built (`:app:assembleDebug`) and committed. These mirror the

## Testing‑feedback fixes

1. **Stop capture during PLAY / PLAY PROCESSED (#2)**
   `CoughCaptureService.playbackActive` flag; capture loop drains but skips analysis while
   set. Set around `ViewerActivity.startPlayback` (covers PLAY and PLAY PROCESSED), cleared
   in `stopAudio`. No more re‑capture of played clips.

2. **Live mic hot‑plug + remember/reacquire chosen mic (#3)**
   `AudioDeviceCallback` handles add+remove via `handleDevicesChanged()`: new mics populate
   the spinner instantly; chosen mic is kept on disconnect and reacquired on reconnect.
   Regression fix: `suppressMicCallback` stops a programmatic spinner rebuild from
   overwriting the persisted `mic_device` with the default.

3. **Gallery: scroll/highlight/flash (#4b)**
   Scroll state preserved across reload; last‑opened recording persisted and highlighted
   (amber); newest acquired recording blinks when visible.

4. **Bluetooth mic capture (#5)**
   BT inputs (SCO/LE) routed via the communication path with a BT‑capable source
   (VOICE_RECOGNITION/MIC, not UNPROCESSED) at 16 kHz/16‑bit, with teardown on stop.
   Routed‑device logging (tag `FFTT04M`, "Capture routed to …") + one‑shot self‑heal of
   the SCO warm‑up race (`btRerouteAttempted`). ⚠️ Unverified on a real device.

5. **Phone‑decided delete after transfer (#4)**
   On receiving the desktop's USB ack, the phone offers to delete the offered recordings
   (+ sidecars) and refreshes the gallery. Explicit Delete/Keep confirm.

## Note on L's Bluetooth
L's BT **data‑transfer** feature (`BtSendActivity` / `BluetoothPairingActivity`) is
socket/RFCOMM‑based and independent of the mic SCO path; manifest BT permissions are
correct (`BLUETOOTH_CONNECT`, `BLUETOOTH_SCAN`, legacy ≤30). If "Bluetooth misbehaves"
refers to file transfer rather than the mic, that needs the device logcat to pinpoint.
